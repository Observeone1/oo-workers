Manage with the CLI

The obs CLI is the canonical interface for self-hosted oo-workers. The dashboard is a real-time observation surface; obs is how you script monitor creation, capture configuration in git, and integrate alert routing into your own playbooks.

Install

npm install -g @observeone/cli

Connect to your master

# Interactive — opens your browser for OAuth/SSO if your master has SSO,
# or prompts for an existing API key.
obs login --host http://localhost:3001

# Or pass a Bearer key directly (good for CI):
OO_KEY=oo_… obs --host http://localhost:3001 monitor list

Create monitors

# HTTP uptime
obs create url --name homepage --url https://example.com

# API check with assertions
obs create api --name payments --url https://api.example.com/health \
  --assert status=200 --assert json:$.ok=true

# TCP / UDP / DB / TLS / Heartbeat — same shape, type-specific flags
obs create tcp       --name pg-reachable --host db.internal --port 5432
obs create db        --name pg-protocol  --host db.internal --port 5432 --protocol postgres
obs create tls       --name edge-cert    --host api.example.com --port 443 --warn-days 30
obs create heartbeat --name nightly-backup --period 60 --grace 30

Capture + restore configuration

# Export everything (config + 90 days of history) to a portable bundle.
obs export --include-scripts > monitors.json

# Apply a bundle on the same or a different instance. Idempotent —
# re-running with the same JSON makes no changes.
obs apply monitors.json

Programmatic API key

# From inside the worker container
docker compose exec worker bun scripts/create-api-key.ts --name ci
# → copy the oo_… key; send it as Authorization: Bearer oo_…

The dashboard's + Add monitor dialog covers the same operations for ad-hoc work. The CLI scales better when you're managing more than a handful of monitors or want them version-controlled.

One row per monitor type with its obs create shape. Click a row's type link to jump to the assertion / configuration details that apply to it.

URL monitor assertions

URL monitors send a HEAD request and check the response status code. One assertion per monitor.

Shape

"assertions": [
  { "operator": "equals", "statusCode": 200 }
]

Operators

Operator Meaning Example
equals HTTP status equals N {"operator":"equals","statusCode":200}
not_equals HTTP status is not N {"operator":"not_equals","statusCode":500}
less_than HTTP status < N {"operator":"less_than","statusCode":400}
greater_than HTTP status > N {"operator":"greater_than","statusCode":199}
No assertions? The check is considered SUCCESS as long as the request didn't error out.

API check assertions

API checks send any HTTP method with headers and body, then evaluate one or more assertions against the response. Multiple assertions allowed; all must pass for SUCCESS.

Shape

{
  "type":     "status_code" | "response_time" | "json_path" | "text_contains" | "header",
  "operator": "equals" | "not_equals" | "less_than" | "greater_than" | "contains" | "not_contains" | "exists",
  "path":     // JSONPath (json_path) or header name (header). Otherwise null.
  "value":    // expected value as string. Numbers are parsed for status_code / response_time.
}

Type × Operator matrix

Type Valid operators path value
status_code equals, not_equals, less_than, greater_than — e.g. "200"
response_time equals, not_equals, less_than, greater_than — ms, e.g. "2000"
text_contains contains, not_contains — substring
header equals, not_equals, contains, not_contains, exists header name (lowercased) expected value
json_path equals, not_equals, contains, not_contains, exists JSONPath, e.g. $.user.email expected value

Worked examples

// Status + speed budget
[
  { "type": "status_code",   "operator": "equals",    "value": "200" },
  { "type": "response_time", "operator": "less_than", "value": "2000" }
]
// JSONPath on a real JSON API
[
  { "type": "status_code", "operator": "equals", "value": "200" },
  { "type": "json_path",   "operator": "equals", "path": "$.userId", "value": "1" },
  { "type": "json_path",   "operator": "exists", "path": "$.title" }
]
// Headers + body content
[
  { "type": "header",        "operator": "contains", "path": "content-type", "value": "application/json" },
  { "type": "text_contains", "operator": "contains", "value": "\"ok\":true" }
]

JSONPath quick reference

JSONPath is a query language for JSON. oo-workers uses the standard jsonpath npm package — same syntax as jq-ish but slightly different.

Expression Meaning
$ The root value
$.user.email Field email nested in user
$.items[0] First element of the items array
$.items[*].id id of every element in items
$..price price anywhere in the document (recursive)
$.items[?(@.in_stock==true)] Filter — only items where in_stock is true
When the assertion's operator is exists, no value is needed — the check passes if the path resolves to anything.

Playwright script reference

Browser checks run any Playwright test file. The script content is stored inline in qa_generated_tests.script; oo-workers writes it to a temp .spec.ts file at run time and invokes npx playwright test. The project's targetUrl is passed in as the PLAYWRIGHT_TARGET_URL env var so scripts don't have to hard-code the URL.

Skeleton

import { test, expect } from '@playwright/test';

const URL = process.env.PLAYWRIGHT_TARGET_URL!;

test('my check', async ({ page }) => {
  await page.goto(URL);
  await expect(page.locator('h1')).toBeVisible();
});

Login flow example

import { test, expect } from '@playwright/test';

test('login + dashboard reachable', async ({ page }) => {
  await page.goto('https://app.example.com/login');
  await page.fill('input[name=email]', 'user@example.com');
  await page.fill('input[name=password]', 'hunter2');
  await page.click('button[type=submit]');
  await expect(page).toHaveURL(/\/dashboard/);
  await expect(page.locator('[data-testid=user-menu]')).toBeVisible();
});

Checkout flow example

import { test, expect } from '@playwright/test';

test('add to cart + checkout', async ({ page }) => {
  await page.goto('https://shop.example.com/p/widget-123');
  await page.click('button:has-text("Add to cart")');
  await page.click('a:has-text("Checkout")');
  await expect(page.locator('.cart-total')).toContainText('$');
});
Each test file launches Chromium headlessly. Keep scripts focused — one or two assertions per file is plenty for monitoring.

Bulk JSON / obs apply

Primary path: obs apply <file.json> from the CLI. The CLI handles the schema, id-anchor remap, and embedded resources for you. Round-trip with obs export --include-scripts > monitors.json for backup or migration between instances. 

# Export from one instance, apply to another
obs --host https://master-a.example.com export --include-scripts > monitors.json
obs --host https://master-b.example.com apply monitors.json

The dashboard's Import JSON button accepts the same payload for one-off imports. The schema below is the wire format the CLI emits — bring your own at your own risk; the CLI handles bindings between monitors, channels, and status pages via id anchors that hand-rolled JSON typically misses.

Schema (v1, advanced reference)

{
  "version": 1,

  "urlMonitors": [
    {
      "name": "my-site",
      "url": "https://example.com",
      "intervalSeconds": 60,
      "timeoutMs": 30000,
      "enabled": true,
      "assertions": [
        { "operator": "equals", "statusCode": 200 }
      ]
    }
  ],

  "apiChecks": [
    {
      "name": "my-api",
      "url": "https://api.example.com/health",
      "method": "GET",
      "headers": { "X-API-Key": "xxx" },
      "intervalSeconds": 60,
      "assertions": [
        { "type": "status_code", "operator": "equals", "value": "200" },
        { "type": "json_path",   "operator": "equals", "path": "$.status", "value": "ok" }
      ]
    }
  ],

  "qaProjects": [
    {
      "name": "login-flow",
      "targetUrl": "https://app.example.com",
      "intervalSeconds": 300,
      "tests": [
        {
          "name": "can-login",
          "script": "import { test, expect } from '@playwright/test'; test('login', async ({ page }) => { ... });"
        }
      ]
    }
  ]
}
Every section is optional — you can import only urlMonitors if that's all you have. Failed items are reported in the response's skipped array, the rest still get created.

Multi-region

Run probes from more than one location by attaching regional agents to this master. Master holds the schedule, fans jobs out per region, and aggregates results. Agents are stateless — they only need outbound HTTPS back to master, no DB, no Redis, no inbound ports.

Architecture

                                  ┌──────────────────────────────┐
                                  │   master (one instance)      │
                                  │  • UI / dashboard            │
                                  │  • scheduler + BullMQ        │
                                  │  • Postgres + Redis          │
                                  │  • /api/agent/{jobs,results} │
                                  └──────┬──────────────────┬────┘
                                         │ outbound HTTPS  │
                       ┌─────────────────┘                 └──────────────┐
                       │                                                  │
                ┌──────▼──────┐                                    ┌──────▼──────┐
                │  us-east    │                                    │  eu-west    │
                │  agent box  │                                    │  agent box  │
                │  (no DB)    │                                    │  (no DB)    │
                └─────────────┘                                    └─────────────┘

Agents long-poll GET /api/agent/jobs?wait=N. Master holds the connection open until a job appears or N seconds pass. Agent runs the probe locally and POSTs the result. Each agent only sees jobs for its own region (authorized via the agent's API key).

Prerequisite: master reachability

The agent box needs to reach this master's HTTP(S) URL. If master binds to 127.0.0.1 (the default), remote agents can't connect. Options:

  • Public HTTPS — front master with a TLS reverse proxy (Caddy auto-LE pairs well).
  • Tailscale / Wireguard — put master and agents on the same private network. Use the master's Tailscale IP or MagicDNS name as OO_MASTER_URL.

Self-signed certs aren't supported by the agent (Bun's fetch validates by default). Use a real cert via Let's Encrypt or a system-installed private CA.

Quick setup (3 steps, ~10 minutes)

1. On master, create the region. Click Regions in the header, fill Slug (lowercase + dashes, e.g. us-east) and Label (e.g. US East (Virginia)). Click Create region. The panel shows the cleartext API key — this is the only time it's displayed. Copy it.

2. On the agent box, fetch the two files and configure:

# Fetch the agent compose + env template.
curl -O https://raw.githubusercontent.com/Observeone1/oo-workers/main/docker-compose.agent.yml
curl -O https://raw.githubusercontent.com/Observeone1/oo-workers/main/.env.agent.example
mv .env.agent.example .env

# Edit .env — set these three values:
#   OO_MASTER_URL   = https://master.example.com
#   OO_AGENT_KEY    = oo_…  (paste from step 1)
#   OO_REGION_SLUG  = us-east

docker compose -f docker-compose.agent.yml up -d
docker compose -f docker-compose.agent.yml logs -f agent

3. Back on master, bind monitors. Refresh the Regions page — the status dot turns green within ~30s. Open + Add monitor, find the "Run from" section, check your region. The scheduler's next tick fans it out.

To bind an existing monitor without re-creating it, use the API: PUT /api/monitors/<type>/<id>/regions with body {"regionIds":[1,2]}. Empty array = run on master only.

What works on agents

Monitor type Agent support
URL, API, TCP, UDP ✅ Full support
Browser (QA / Playwright) ❌ Returns ERROR — runs only on master

Browser monitors need a heavy Chromium runtime that isn't yet shipped on agents. If you bind a QA monitor to a region, the agent reports each run as ERROR with a clear message. Uncheck the region in the dialog (or delete from monitor_regions) to run it on master only.

Multiple agents per region

Safe — Redis BRPOP guarantees only one agent receives each job, and master's result write is idempotent on executionId. You get parallel probe throughput for free. Each agent box uses the same agent key.

Rotating an agent key

From the Regions page: click Rotate key on the row. A new cleartext key panel appears; copy it. The running agent starts getting 401 — restart it with the new env vars and it picks up where it left off. Region history is preserved.

CLI equivalent (atomic):

docker compose exec worker bun scripts/rotate-region-key.ts --slug us-east

Deleting a region

From the Regions page: click Delete. This revokes the agent's API key and removes all monitor_regions bindings (cascading). Existing execution history is preserved — region_id on those rows is set to NULL.

Stalled executions

If an agent crashes mid-probe, the corresponding execution row stays at PENDING. Master keeps scheduling new probes every interval, so the metric stays fresh — only the stale row leaks. The dashboard and API automatically project these as FAILED once they're older than 2× the monitor's interval. There's no background sweeper; the projection happens at read time. A late agent result can still write into the row.

Preflight

Before starting the agent, run the connectivity check inside the agent container. It catches DNS / TLS / auth / region-slug mistakes up front and exits non-zero on the first failure: 

docker compose -f docker-compose.agent.yml run --rm agent \
  bun scripts/check-agent-connectivity.ts

Reads OO_MASTER_URL / OO_AGENT_KEY / OO_REGION_SLUG from the agent's env. Reports each step with ✅ / ❌.

Troubleshooting

Agent log says… Likely cause
ECONNREFUSED or DNS errors Agent can't reach OO_MASTER_URL. Master is probably bound to 127.0.0.1, or DNS for the master hostname is wrong. Expose master publicly with TLS or put both on the same VPN.
401 invalid or revoked agent key Key was rotated/revoked, or you mis-pasted the cleartext. Rotate from the Regions page and re-paste.
socket connection closed unexpectedly Should not happen with master at v0.7.0+ (idleTimeout was bumped to 120s). If you see it, confirm master is on the latest tag.
Region offline in UI but agent log looks healthy Refresh the Regions page; the online threshold is 60s of inactivity.
After every master version bump, run docker compose -f docker-compose.agent.yml pull && docker compose -f docker-compose.agent.yml up -d on each agent box. Agents share the master's image, so version drift between them is worth avoiding.
